# Security

## 지원 버전

Parism은 v1.0.0 안정 릴리스 단계다. 보안 패치는 최신 릴리스에 제공된다. 이전 마이너 버전에 대한 백포트는 제공하지 않는다. 보안 수정이 포함된 새 버전이 출시되면 즉시 업그레이드할 것을 권고한다.

| 버전 | 지원 여부 |
|------|----------|
| 1.x (latest) | 보안 패치 제공 |
| 0.x | 지원 종료 |

---

## 취약점 신고

보안 취약점은 GitHub Security Advisories를 통해 비공개로 신고한다.

신고 URL: https://github.com/JinHo-von-Choi/parism/security/advisories/new

신고 후 처리 절차:

1. 수신 확인 — 며칠 이내에 접수 확인 응답
2. 초기 평가 — 재현 가능 여부 및 영향 범위 분석
3. 수정 및 릴리스 — 심각도에 따라 우선순위 결정 후 패치 릴리스
4. 공개 — 패치 배포 후 적절한 시점에 CVE 또는 advisory 공개

공개 이슈 트래커(GitHub Issues)에 보안 취약점을 보고하지 말 것. 악용 가능성이 있는 세부 내용이 패치 전에 노출될 수 있다.

## 위협 모델

### Parism이 방어하는 것

Parism의 Guard는 에이전트가 생성한 명령이 시스템에 예상치 못한 범위로 실행되는 것을 막는 가드 수준 방어선이다.

- allowed_commands 화이트리스트에 없는 명령은 프로세스가 생성되기 전에 차단된다.
- allowed_paths 바깥의 경로로 향하는 인자는 실행 전에 거절된다.
- `;`, `$(`, 백틱, `&&`, `||`, `|`, `>`, `>>`, `<` 등 셸 인젝션 패턴이 인자에 포함되면 실행하지 않는다.
- `node -e`, `node --eval`, `npx --yes`, `curl -o` 등 명령별 위험 플래그를 차단한다.

Parism은 `execFile`을 사용하여 프로세스를 직접 실행한다. 셸을 거치지 않기 때문에 `;`, `$()`, 백틱, 파이프를 인자로 전달해도 셸 인터프리터가 해석하지 않는다. 이 구조적 특성만으로 셸 인젝션의 대부분은 스폰 레이어에서 원천 차단된다.

### Parism이 방어하지 않는 것

Guard는 가드 레이어 방어선이며 커널 샌드박스가 아니다. 다음 위협은 Guard의 범위 밖이다.

- 화이트리스트에 등록된 바이너리 자체의 악의적 동작 — 예를 들어 `rm`이 allowed_commands에 있고 allowed_paths 안쪽 경로로 호출되면, 해당 경로의 파일 삭제를 막지 않는다.
- 실행되는 바이너리에 포함된 커널 수준 익스플로잇 또는 컨테이너 탈출
- allowed_commands에 등록된 명령의 공급망 침해 (바이너리 자체가 악성인 경우)
- 타이밍 사이드 채널, 파일 잠금 경합 등 부채널 공격
- Node.js 프로세스 메모리를 직접 조작하는 공격

## 4겹 방어선

### (a) allowed_commands 화이트리스트

`allowed_commands`에 등록되지 않은 명령어는 프로세스를 생성하지 않는다. 차단 시 `failure.kind: "guard"`, `failure.reason: "command_not_allowed"` 응답이 반환된다. 하위 호환 필드 `guard_error.reason` 도 함께 emit 된다. 권위 필드는 `failure` 이며, 새 소비자는 이쪽을 사용한다.

한계: 화이트리스트 자체의 범위가 너무 넓으면 (예: `bash`, `sh`, `python` 등) 이 계층의 방어 효과가 크게 줄어든다. 화이트리스트는 실제로 필요한 명령만 포함해야 한다.

### (b) allowed_paths 경로 제한

`allowed_paths`를 설정하면 `cwd`와 경로 인자를 검사한다. `/`, `./`, `../`로 시작하는 인자와, `cat`, `find`, `ls`, `grep`, `git`, `docker`, `kubectl`, `cargo` 등 경로를 받는 명령의 positional 인자가 허용 경로 밖이면 차단된다.

한계: 커널 레벨 강제가 아니므로, 경로를 직접 받지 않는 명령(예: `env`, `id`, `uname`)에는 경로 검사가 적용되지 않는다. `allowed_paths`가 비어 있으면 이 계층의 검사가 생략된다.

### (c) 인젝션 패턴 차단

인자 배열을 개별 인자 단위로 순회하며 `;`, `$(`, 백틱, `&&`, `||`, `|`, `>`, `>>`, `<`를 검사한다. 해당 패턴이 포함된 인자가 있으면 실행하지 않는다. 인자 단위 검사이므로 서로 다른 인자의 경계를 넘어서는 오탐이 발생하지 않는다.

한계: 이 검사는 인자 레벨에서 동작하며, execFile의 구조적 셸 분리를 보완한다. 바이너리가 인자를 내부적으로 셸에 위임하는 드문 경우(예: 일부 스크립트 래퍼)까지 막지는 못한다.

### (d) 명령별 인자 제한 (command_arg_restrictions)

명령마다 차단할 플래그를 정의한다. 기본 차단 플래그:

- `node`: `-e`, `--eval`, `-r`, `--require`, `-p`, `--print`, `--input-type`
- `npx`: `--yes`, `-y`
- `curl`: `-o`, `--output`

한계: `command_arg_restrictions`에 정의되지 않은 플래그는 검사하지 않는다. 새 명령을 화이트리스트에 추가할 때 위험 플래그를 함께 정의해야 한다.

## 신뢰할 수 없는 입력에 대한 권고

Parism을 신뢰할 수 없는 외부 호출자에게 노출할 경우(예: 공개 API 게이트웨이, 다중 테넌트 환경), Guard 단독으로는 충분하지 않다. 다음을 권고한다.

컨테이너 또는 VM 격리: Parism 인스턴스를 격리된 컨테이너나 VM 안에서 실행한다. Guard가 뚫리더라도 호스트 시스템으로의 피해 확산을 막는 마지막 방어선이 된다.

최소 권한 원칙: Parism 프로세스가 실행되는 OS 사용자 계정의 권한을 최소화한다. 루트로 실행하지 말 것.

allowed_commands 최소화: 실제로 에이전트가 사용하는 명령만 화이트리스트에 포함한다. `bash`, `sh`, `python`, `perl` 등 범용 인터프리터는 절대 포함하지 않는다.

allowed_paths 명시: `allowed_paths`를 비워두면 경로 제한이 없다. 작업 범위가 명확하다면 반드시 설정한다.

### 커스텀 파서 샌드박스 (Phase 2 예정)

`~/.parism/parsers/`에 등록된 외부 ParserPack 파일은 현재 Node.js 프로세스 컨텍스트에서 직접 로드된다. `vm.runInContext` 기반 격리 정책은 Phase 2 ADR에서 논의 중이며 아직 적용되지 않았다.

신뢰할 수 없는 제3자 파서를 로드할 계획이라면, 그 전까지 Parism 자체를 컨테이너 또는 VM 안에서 실행할 것을 강력히 권고한다.

### 출력 리댁션 (Output Redaction)

`guard.secrets.output_redaction_enabled` 옵션은 Phase 2 기능으로 기본값은 비활성화다. 활성화하면 파싱이 끝난 raw/stderr 출력에서 공통 시크릿 패턴(토큰, 비밀번호 등)을 마스킹한다. 파싱 전에 원본을 변조하지는 않는다.

출력 리댁션은 옵트인이며 현재 기본 설정에서는 동작하지 않는다. 시크릿이 stdout에 노출될 가능성이 있는 환경에서는 이 옵션 외에도 `env_secret_patterns` 설정으로 자식 프로세스에서 환경 변수를 사전 제거할 것을 권고한다.
